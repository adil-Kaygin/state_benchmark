# Brainstorm — Lightweight learned filter, UKF-robust at EKF cost (2026-06-29)

## Goal

A learned state estimator that is **as cheap as EKF at inference** (no sigma-point
propagation, no matrix inversions, one trajectory / one timestep at a time on CPU)
but **as robust as UKF / advanced filters on strongly nonlinear systems**.

Same premise as the existing `kalmannet`: we are given the *true* model `f`, `h`
(no model mismatch) — same `f`/`h` that EKF/UKF get. We are free to use or ignore
`Q`, `R`. EKF/UKF compute the gain analytically from `P`, `Q`, `R`, Jacobians;
the learned filter replaces that machinery with a network.

## Why EKF-class methods fail on strong nonlinearity (the thing to beat)

The current KalmanNet update is **affine in the innovation**:

```
x_pred = f(x_prev)
innov  = y - h(x_pred)
x_post = x_pred + K . innov          # K from GRU, but correction still affine
```

EKF has the *same* structural limit — a single gain matrix maps innovation ->
correction. That linearization is exactly why EKF degrades under curvature.
UKF's edge is different in kind: it propagates the *distribution* through the true
nonlinear `f`/`h` (sigma points), capturing curvature no single gain can express.

So: a network that emits a free `K` adapts the gain better than EKF, but cannot
escape the affine-update form. To rival UKF cheaply we must **represent curvature
without propagating sigma points.**

## Design axes (these are orthogonal — don't bundle them)

1. **Update form**: affine `K.innov`  vs.  **nonlinear** `MLP([innov, x_pred, ...])`.
   -> This is THE lever for nonlinearity.
2. **Memory**: full GRU (recurrent implicit covariance)  vs.  windowed innovations
   vs.  one running statistic (EMA)  vs.  none (memoryless).
   -> This is the "trust adapts over time" lever; EKF's `P_t` lives here.
3. **Local geometry inputs**: none  vs.  Jacobian `H = dh/dx|x_pred`  vs.  cheap
   finite-difference curvature probe ("poor man's sigma points").
4. **Q/R usage**: ignore (current premise)  vs.  reuse for an EKF-gain regularizer.

The user's instinct (drop affine update, keep EKF-style innovation) = axis 1.
That is correct and is the highest-leverage single change.

## Candidate architectures (cheapest first)

### V0 — Memoryless nonlinear update (the user's literal idea, fixed)

```
x_pred = f(x_prev)
innov  = y - h(x_pred)
x_post = x_pred + MLP([innov, x_pred])     # x_pred input is NOT optional
```

- **Why `x_pred` must be an input:** in a nonlinear system the same innovation
  means different corrections at different states (EKF re-linearizes `H` at
  `x_pred` every step). `MLP(innov)` alone learns ONE global map -> strictly less
  adaptive than EKF. Adding `x_pred` restores state-dependence.
- **Cost:** cheapest possible — one small MLP per step. No covariance, no Jacobian.
- **Limit:** no temporal memory. Same `(innov, x_pred)` gives same correction at
  t=2 and t=200 -> no "correct hard early, gently once converged" behavior.
- **Verdict:** excellent *baseline*. So cheap it must be measured. Likely beats EKF
  on per-step nonlinearity, likely loses on transients / partial observability.

### V1 — Recurrent state + nonlinear update (RECOMMENDED starting point)

```
h_t    = recurrence(h_{t-1}, [innov, dx_prev])    # keep memory (GRU-lite)
x_post = x_pred + MLP([innov, x_pred, h_t])        # nonlinear, not K.innov
```

- Keeps the GRU's role as implicit covariance / trust tracker (axis 2) AND breaks
  the affine update (axis 1). This is the smallest change to the current code that
  targets the actual nonlinearity limit.
- Still EKF-cheap: one small recurrence + one small MLP per step. No sigma points,
  no inversions.
- Slots into the existing `_KalmanGainGRU` (add an MLP correction head; change the
  correction line in the three forward paths) — see "Code touch-points" below.

### V2 — Curvature-aware inputs ("poor man's UKF")

Add cheap local curvature features to V1's MLP without UKF's 2nx+1 propagations:

```
y0 = h(x_pred)
y± = h(x_pred ± delta * e_i)     # a few directional probes, or random delta
curv_feats = [y+ - 2*y0 + y-]    # second-difference ~ local curvature of h
x_post = x_pred + MLP([innov, x_pred, h_t, curv_feats])
```

- The single biggest reason UKF beats EKF is curvature in `h` (and `f`). A handful
  of finite-difference probes captures the dominant curvature for `< 2nx+1` model
  evals — and we can probe only the directions that matter (e.g. along innovation).
- Jacobian `H` can be one of these features too, but note `H` is *first-order*; the
  second-difference probe is what actually adds UKF-like information.
- Cost scales with number of probes (each = one `h` eval). Tunable knob between
  "EKF-cheap" and "UKF-cheap".

### V3 — Differentiable EKF / structured gain head (most principled, biggest change)

GRU emits `Q`/`R` (or a `P`-increment, or a multiplicative correction to `K_ekf`);
`K` is then computed through the *real* EKF formula using the Jacobian `H`. Update
stays affine but the gain is model-consistent and learned. This is "deep EKF",
NOT what closes the nonlinearity gap (still affine) — listed for completeness.
Lower priority for this goal.

## On Jacobians specifically (the earlier question)

- As **MLP inputs**: useful but first-order; makes the net more EKF-like. Helpful,
  not sufficient. The finite-difference curvature probe (V2) is the better use of
  the same model-eval budget.
- As a **custom loss** term: the defensible form for THIS goal is a *difficulty
  weight* — upweight timesteps with large innovation / fast-changing `||H||`
  (where EKF would fail) so the net spends capacity there. The EKF-gain-matching
  regularizer (`||K_gru - K_ekf||^2`) pulls *toward* EKF — wrong direction here.

## Loss ideas (beyond plain MSE)

- **State MSE** (baseline, current).
- **Curvature-weighted MSE**: `w_t = 1 + lambda * curvature(x_pred_t)`; focus
  learning where linearization fails.
- **Measurement-consistency aux term**: `||y_t - h(x_post_t)||^2` — cheap, no
  Jacobian, keeps corrections self-consistent in measurement space.
- **Innovation-whiteness penalty**: penalize correlation of successive innovations
  (optimal filters produce white innovations) — encourages the recurrent state to
  behave like a real covariance.

## Recommendation / suggested order of experiments

1. **V0** memoryless `MLP([innov, x_pred])` — trivial, sets the floor. (1 small class)
2. **V1** recurrent + nonlinear MLP update — expected best cost/benefit. Measure vs
   EKF and UKF on the nonlinear level(s).
3. If V1 still trails UKF on the hardest level: add **V2** curvature probes (start
   with probes along the innovation direction only — 2 extra `h` evals).
4. Loss: start MSE; add curvature-weighting only if V1/V2 underfit the hard regime.

Keep all of this as a **separate estimator class** (e.g. `nonlinear_update` /
`deep_filter`) so the existing `kalmannet` is undisturbed. Reuse the same
hardware-split contract: batched torch `f`/`h` on GPU for training
(`FilterModel.torch`), strictly-sequential NumPy `f`/`h` on CPU for `estimate()`
(microprocessor-style), exactly as `kalmannet.py` does today.

## Code touch-points (for when we implement)

- New class mirrors `KalmanNetEstimator`'s three forward paths:
  - GPU free-running: `_run_sequence_vectorized` (replace the `K @ innov`
    correction at kalmannet.py:325-326 with the MLP head).
  - Teacher-forced parallel: `_run_sequence_teacher_forced`
    (kalmannet.py:422-426).
  - CPU sequential inference: `_run_sequence_sequential_cpu`
    (kalmannet.py:723-725) — this is where EKF-cost parity is proven.
- Prediction/innovation lines are UNCHANGED from kalmannet (use `f`, `h` exactly
  as EKF does: ekf.py uses the same `nd.f`/`nd.h`).
- For V3 only: would need the Jacobians `F_jac`/`H_jac` that EKF already consumes
  (ekf.py:61-62), so they're available per level.

## Open questions

- How many curvature probes (V2) before inference cost stops being "EKF-like"?
- Does memory (V1) actually beat memoryless (V0) on OUR levels, or are they
  fast-mixing / fully-observable enough that V0 suffices? -> empirical, cheap to test.
- Is the win (if any) from the nonlinear update, the curvature inputs, or the
  memory? Ablate by building V0 -> V1 -> V2 incrementally.
