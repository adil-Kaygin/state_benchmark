# Brainstorm — neural-training profiling results & next steps

Working notes on what [`notebooks/profile_neural_training.ipynb`](notebooks/profile_neural_training.ipynb)
measured (Issue 14) and what to do about it. This is a scratch/decision doc, not a
spec — findings feed back into Issues 9/11/12/13. Each item is tagged:

- **[confirmed]** — the profiler validated a diagnosis from an issue.
- **[problem]** — a real inefficiency or a gap the numbers expose.
- **[caveat]** — a limitation of the measurement itself (trust the shape, not the
  absolute number).
- **[recommendation]** — a concrete next action.

The run captured here was on the **reduced Lorenz** config (90 train traj, T=200,
batch=32, 3 epochs) on `torch 2.11.0+cu128`, CUDA available. **The run is
incomplete**: Cell 2's CPU sweep was cancelled mid-Transformer, so **Mamba-CPU is
missing** and there is no CPU `torch.profiler` table.

---

## 1. Sequential-recurrence estimators are faster on CPU than GPU — [confirmed]

Cell 2 headline (reduced Lorenz, total fit() seconds):

| Estimator   | GPU total | CPU total | Winner                |
|-------------|-----------|-----------|-----------------------|
| NeuralODE   | 46.1s     | 36.7s     | **CPU** (1.26×)       |
| PINN        | 13.6s     | 8.4s      | **CPU** (1.62×)       |
| KalmanNet   | 8.7s      | 5.7s      | **CPU** (1.53×)       |
| Mamba       | 6.3s      | *(cancelled)* | —                 |
| Transformer | 0.9s      | 18.9s     | **GPU** (21×)         |

This is the core Issue 12 thesis validated: for the estimators with an irreducible
sequential T-loop (NeuralODE, PINN, KalmanNet), the CPU wins because each tiny
per-step op pays full kernel-launch latency while the GPU is idle. Transformer —
non-sequential, matmul-heavy — shows the *opposite* (GPU 21×), which is the
contrast that makes the launch-bound argument credible rather than a blanket "CPU
is faster."

**[recommendation]** Keep KalmanNet `phase2_device=None → CPU` as the default
(Issue 12 Lever 1). For PINN and NeuralODE, wire the per-(level, estimator) config
to allow `device="cpu"` on short/narrow sequential levels like Lorenz, and document
"CPU may be faster for short/narrow sequential levels" near the configs.

## 2. Launch-bound / GPU-idle signature is proven — [confirmed]

Cell 7 `torch.profiler`, one NeuralODE forward+backward (n_substeps=8):

- ~**19,800** `AddmmBackward0` calls, ~**39,600** `aten::mm`, ~**56,000**
  elementwise-kernel launches.
- Per-kernel CUDA time ~3–15µs (tiny).
- **Self CUDA total 1.31s vs Self CPU total 6.65s.**

The gap between CPU dispatch time (6.65s) and actual GPU compute (1.31s) *is* the
idle GPU — the launch-bound fingerprint Issue 12 predicted. Op counts scale with
`T × n_substeps × 4`, tying the cost directly to the RK4 inner loop.

**[recommendation]** No new action — this is the "definitive evidence" the notebook
set out to produce. Use this table as the artifact in the Issue 12 PR description.

## 3. NeuralODE is the worst offender and is linear in n_substeps — [confirmed]

Cell 6 (NeuralODE, s/epoch):

| n_substeps | s/epoch | drift evals/traj (T·ns·4) | ratio vs ns=2 |
|------------|---------|---------------------------|---------------|
| 2          | 4.24    | 1600                      | 1.00×         |
| 4          | 8.21    | 3200                      | 1.94×         |
| 8          | 15.97   | 6400                      | 3.76×         |

Near-perfectly linear, matching the drift-eval count. Confirms Issue 11's target
(hoist the per-eval `t_col` allocation out of the 4·ns·T inner loop) is the right
hot path, and that `n_substeps` is the single biggest cost knob for this estimator.

**[recommendation]** In the real config, treat `n_substeps` as an accuracy/cost
dial and pick the smallest value that holds RMSE. The `t_col` hoist (Issue 11)
removes ~3000 tiny allocations/batch but does **not** change the linear scaling —
manage `n_substeps` too.

## 4. Backward dominates; opt.step is negligible — [confirmed]

Cell 3 forward / backward / opt.step split (GPU, seconds):

| Estimator   | forward | backward | opt.step |
|-------------|---------|----------|----------|
| NeuralODE   | 1.67    | 2.67     | 0.007    |
| PINN        | 0.57    | 0.89     | 0.003    |
| KalmanNet   | 0.75    | 0.81     | 0.111    |
| Mamba       | 0.55    | 0.47     | 0.005    |
| Transformer | 0.31    | 0.10     | 0.005    |

Backward > forward for every sequential net (the autograd graph replays the same
launch-bound loop in reverse). opt.step is ≤0.1s everywhere → optimizer choice is
not the bottleneck. Transformer inverts (forward > backward), consistent with being
compute-bound.

**[recommendation]** Optimization effort belongs on the forward loop structure
(fewer/bigger kernels), not on the optimizer or the step. Don't spend time on
opt.step micro-tuning.

## 5. Batch-size sweep is confounded by update-count starvation — [problem] [caveat]

Cell 5 (KalmanNet, GPU, s/epoch):

| batch | s/epoch | batches/epoch |
|-------|---------|---------------|
| 32    | 3.06    | 2             |
| 64    | 1.89    | 1             |
| 128   | 1.07    | 1             |

The speedup is real but **partly comes from doing fewer optimizer steps**, not just
from amortizing launch overhead — exactly the trap Issue 12 Lever 2 warns about. On
90 train traj, batch 128 = 1 batch/epoch = the model barely trains. So this cell
demonstrates the *mechanism* but must **not** be read as "just use batch 128."

**[recommendation]**
- Re-run this sweep on a config with enough trajectories that batches/epoch stays
  ≥ ~4 across the sweep, so launch-amortization is isolated from update count.
- For the real Lorenz config, follow Issue 12: a **modest** 32→64 (maybe 96) bump
  with a linear LR scale, RMSE confirmed non-regressed. Do **not** crank to 256+.

## 6. Phase-1 vs Phase-2 wall-clock is never actually isolated — [problem] [fixed]

Cell 4's markdown promised a "per-epoch wall-clock split by phase," but the cell
only printed total fit time (6.5s) and the phase *counts* (1 Phase-1 epoch, 3
Phase-2). It never reported "P1 s/epoch vs P2 s/epoch" — the one number the cell
exists to produce. It also exposed an off-by-one in the reduced config:
`curriculum_epochs = max(3//2,1) = 1` Phase-1 epoch **plus** `num_epochs=3`
Phase-2 epochs = 4 total, so PROF_EPOCHS no longer means total epochs here.

Side finding [confirmed, Issue 13]: the loss trace shows the exposure-bias jump —
Phase-1 val 1.21 → Phase-2 val 14.0 → 11.7 as it switches teacher-forced →
free-running.

**[fixed]** Cell 4 now wraps the two per-epoch entry points
(`_run_epoch_phase1_cached`, `_run_epoch`) in a wall-clock timer that syncs
against each phase's own device (Phase 1 → training device, Phase 2 →
`phase2_device`, CPU by default) and reports **s/epoch per phase** plus the P2/P1
ratio — the number that motivates running Phase 2 on CPU. This is a pure
measurement wrap (no estimator code changed, per the notebook's contract). The
cell's markdown now states the epoch bookkeeping explicitly: total epochs =
`curriculum_epochs + num_epochs`, so PROF_EPOCHS is *not* the total, and s/epoch
is reported precisely because it's invariant to those counts.

## 7. No baseline → the "speedup" is unquantified — [problem] [caveat]

Every number here is the **post-fix** state (branch
`perf/neural-training-speedups-issues-9-11-12-14`). The notebook parameterizes
`BRANCH`, but no `main` run is captured, so we cannot state how much Issues 9/11/12
actually saved — only that the *shape* matches the diagnoses. The issues' acceptance
criteria explicitly ask for before/after wall-clock.

**[recommendation]** Run the same notebook against `BRANCH="main"` and diff Cell 2 /
Cell 6 against this run. Record the before/after delta in each PR description. This
is the single highest-value follow-up.

## 8. Measurement trust boundaries — [caveat]

- **Reduced config**: 90 train traj, T=200, 3 epochs. Relative costs and the launch
  pattern are trustworthy; **absolute** seconds are not representative of the real
  2000-traj / 400-step / 100-epoch run.
- **Incomplete run**: Mamba-CPU missing; no CPU `torch.profiler` table (so the
  "op count still surfaces on CPU" claim in Cell 7's markdown is untested here).
- **Sync discipline is correct**: `_sync()` calls `cuda.synchronize()` around every
  timer, so GPU numbers measure compute, not dispatch — the timings are sound as far
  as they go.

**[recommendation]** Re-run Cell 2 to completion (fill in Mamba-CPU), and once on a
config nearer the real one to sanity-check that the CPU-wins-for-sequential ordering
holds at scale (larger batches shift the CPU/GPU crossover — the escape-hatch
`phase2_device="cuda"` exists for exactly that).
